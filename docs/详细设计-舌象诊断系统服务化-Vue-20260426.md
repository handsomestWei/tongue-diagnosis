# 舌象诊断系统 — 服务化与平台化详细设计

**文档类型**：详细设计（Detailed Design）  
**技术约束**：默认 CPU-only；前端采用 **Vue 3** 技术栈  
**版本**：v1.2  
**日期**：2026-04-26（修订：账号登录与 RBAC）  

---

## 1. 文档说明

### 1.1 目的

在现有仓库（`tongue_diag`、`tongue_sam`、`labelme_yolo` 等）基础上，建设**系统化、可服务化交付**的舌象诊断平台，满足：

- 依赖向 **CPU-only** 收敛；
- 对外 **HTTP API** 供第三方系统集成；
- **Vue** 前端覆盖数据上传、人工标注、训练、增量训练、推理与纠错闭环；
- 代码结构引入 **`core/`** 层级承载核心域逻辑；
- 数据层支持 **SQLite3** 与 **MySQL** 可配置切换。

### 1.2 范围

- **在仓**：后端服务（FastAPI 建议）、Vue 前端、数据访问与迁移、与现有推理/训练链路的编排封装。
- **不在本文展开**：具体 UI 视觉规范细稿、生产环境网络与安全等保细节（可在部署文档中补充）。

### 1.3 术语

| 术语 | 说明 |
|------|------|
| core | 核心域模块目录，不依赖 Web 框架，可被 API/Worker/CLI 复用 |
| API 模块 | 对外 HTTP 接口层（路由、鉴权、DTO） |
| Worker | 长耗时任务消费者（训练、大批量推理） |
| 纠错闭环 | 推理结果 → 人工纠正标签 → 写库 → 可纳入后续训练 |
| 输入图形态（image_kind） | 全脸自拍需 SAM 抠舌；舌部特写跳过 SAM，整图即舌区 |
| 训练用规范图 | 无论何种输入，**进入 YOLO classify 的须为统一尺寸的舌部 ROI 图**（由管线生成或等价于原图） |

## 5. 核心域（core）设计要点

### 5.1 职责边界

- **core**：纯业务与算法编排；**禁止**直接依赖 FastAPI 的 `Request`/`Depends`。
- **api**：HTTP 映射、认证、分页 DTO、错误码。

### 5.2 `PreprocessForClassify`（核心服务抽象）

建议在 **core** 中实现独立模块（名称示意）：根据 `image_kind` 调度 **SAM 路径** 或 **特写直通路径**，输出统一格式的 `np.ndarray` / 临时文件路径，供分类器与训练导出共用。

### 5.3 输入图两种形态（业务必选能力）

系统须显式区分并贯穿 **上传 → 预处理 → 训练导出 → 推理** 的输入类型（字段建议：`image_kind`，枚举见下）。**不允许**在未声明类型时默认定论为某一种，以免错误跳过 SAM 或错误对特写图做全脸分割。

| 枚举值（建议） | 用户场景 | 预处理（进入分类器之前） | 说明 |
|----------------|----------|---------------------------|------|
| `full_face_selfie` | 手机自拍 **含全脸** | **必须**走 TongueSAM（+ 既有 YOLOX 提示）→ 舌部 mask/框 → 裁切与 letterbox/统一尺寸 | 与现有 `tongue_sam` + ROI 流水线一致 |
| `tongue_closeup` | **仅拍到舌头区域**（特写） | **跳过 SAM**；原图经统一 resize/letterbox（及可选 CLAHE 等与 ROI 后处理对齐的增强）后 **直接送分类器** | 避免无意义分割与失败风险 |

**设计原则**：

- **训练与推理对齐**：同一 `image_kind` 在训练集构建与在线推理中走**同一条** `PreprocessForClassify` 逻辑，避免 train/serve skew。
- **训练用「规范舌图」**：导出到 `train/类名/*`、`val/类名/*` 的文件，一律为**已定宽高的舌部 ROI 图像**；对 `tongue_closeup` 而言，规范图即「整幅特写图按与 full_face 相同的后处理参数导出的结果」。
- **人工修正**：标注界面可提供「改判输入类型」能力（审计写入 `Label` 或 `Image`），用于纠正上传时误选；修正后应**可触发重新生成规范图**（异步任务）。

**可选扩展（后续）**：`image_kind=auto` 由模型或启发式判定再分流——**不在 v1 必达**，若做需独立评审（误判代价高）。

### 5.4 推理 / 训练统一管线（按 image_kind 分支）

```text
输入 bytes / 文件路径 + image_kind
       |
       +-- full_face_selfie --> TongueSAM + ROI 裁剪 --> 规范舌图 tensor/array
       |
       +-- tongue_closeup ----> 仅几何/色彩规范化 ------------> 规范舌图 tensor/array
                                   |
                                   v
                        Ultralytics YOLO classify（CPU）
```

全流程默认 **CPU**；`full_face_selfie` 路径耗时更长，API 层应对该枚举单独 **超时 / 队列配额** 可配置。

### 5.5 TongueSAM 与 `predict.py`（与 §5.3 的关系）

- **问题**：当前 `tongue_sam/predict.py` 顶层即加载模型，不利于服务进程复用与测试。
- **目标**：抽成 `run_inference(...)`（或类 `TongueSamEngine`），支持惰性加载、单次 batch、显式 `device="cpu"`；**仅**在 `image_kind=full_face_selfie` 时调用。
- **过渡**：第一阶段可用子进程调用现有脚本；第二阶段改为进程内调用以减少启动开销。

### 5.6 训练与增量训练

- **全量训练**：从 DB 按样本的 `image_kind` **逐条执行 §5.4 分支**，生成规范舌图后，再导出为 `train/类名/*`、`val/类名/*`，调用 Ultralytics `YOLO.train(..., device="cpu")`。同一训练任务内**允许混合**两种 `image_kind` 的样本。
- **增量训练**：记录 `ModelRegistry.parent_model_id`；数据集 = 父版本快照 + 新增已标注图片 + **纠错表**中标记为「参与再训练」的样本（策略可配置：全量合并 / 仅错题）；导出时**同样**按 `image_kind` 走预处理。

## 6. 数据存储设计

### 6.1 配置

- 单一 **`DATABASE_URL`**（12-factor）：
  - SQLite：`sqlite:///./data/app.db`
  - MySQL：`mysql+pymysql://user:pass@host:3306/db?charset=utf8mb4`

### 6.2 技术栈

- **SQLAlchemy 2.0** + **Alembic** 迁移。
- Repository 模式：业务服务依赖抽象接口，便于单测 mock。

### 6.3 核心表（概念 ER）

| 实体 | 说明 |
|------|------|
| User | **必选**；唯一 `username`、**密码哈希**（如 bcrypt）、**`role`**（`admin` \| `annotator` \| `viewer`）、`is_active`、`last_login_at`；演示阶段可为内存/种子账号，生产必须落库 |
| Project | 多项目隔离（可先单默认项目） |
| Image | 原图元数据、存储路径、hash、**`image_kind`**（`full_face_selfie` \| `tongue_closeup`）、可选 **`derived_tongue_path`**（规范舌图缓存路径）、**`preprocess_version`**（几何/增强参数版本，避免训练漂移） |
| Label | image_id、class_name、source(manual/model)、confidence |
| ModelRegistry | 权重路径、框架、metrics_json、parent_model_id、status |
| TrainJob | status、params_json、日志、起止时间 |
| Prediction | image_id、model_id、result_json、latency_ms |
| Correction | prediction_id、correct_class、remark、include_in_next_train |

**文件存储**：DB 仅存相对 `STORAGE_ROOT` 的路径；大文件不进入版本库。

### 6.4 SQLite vs MySQL 差异处理

- 避免依赖 MySQL 独有特性于核心路径；JSON 可用 JSON 类型（SQLAlchemy 兼容层）或 Text 存序列化。
- 高并发写场景：**推荐 MySQL**；SQLite 适合开发与单机 MVP。

## 7. API 模块设计

### 7.1 规范

- REST + JSON；**OpenAPI 3** 自动生成（`/docs`、`/redoc`）。
- 错误体统一：`{"error": {"code": "...", "message": "...", "detail": ...}}`。

### 7.2 认证与安全

| 场景 | 机制 | 说明 |
|------|------|------|
| **Web 登录** | **OAuth2 Password** → **JWT（Bearer）** | `POST /api/v1/auth/login`，body：`application/x-www-form-urlencoded`（`username`、`password`）；返回 `access_token`（短期，默认 24h 可配置）、`role`、`username` 等 |
| **Web 请求** | `Authorization: Bearer <access_token>` |  axios/fetch 全局拦截；**401** 清会话跳转登录；**403** 提示权限不足 |
| **系统集成** | **API Key**（可选并存） | Header `X-API-Key`；与 JWT 二选一或分路由（machine-to-machine 推荐 Key） |
| **密码存储** | **bcrypt**（或 Argon2） | 禁止明文；实现上使用 `bcrypt` 库；生产强制首次改密策略可后续增加 |

**JWT 载荷（建议）**：`sub`（username）、`role`、`exp`；**刷新令牌**可在 v2 增加（当前可先单 access token）。

### 7.3 角色与权限矩阵（RBAC）

| 能力 / 路由 | admin | annotator | viewer |
|-------------|:-----:|:---------:|:------:|
| 仪表盘 `/` | ✓ | ✓ | ✓ |
| 批量上传、标注 | ✓ | ✓ | ✗ |
| 全量/增量训练 | ✓ | ✓ | ✗ |
| 批量推理、纠错审核 | ✓ | ✓ | ✓（viewer 建议**只读**：不提交纠错/不上传，仅查看；前端禁用 + 后端校验） |
| 模型注册/默认模型 | ✓ | ✗ | ✗ |
| 系统设置 | ✓ | ✗ | ✗ |

**原则**：前端路由 `meta.allowRoles` 做体验拦截；**后端每个 mutating 接口**必须用 `Depends(require_roles(...))` 或等价装饰器校验，防止绕过前端。

### 7.4 端点（初版清单）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /health | 健康检查（无鉴权） |
| POST | /api/v1/auth/login | 用户名密码登录，返回 JWT |
| GET | /api/v1/auth/me | 当前用户（Bearer） |
| GET | /api/v1/models | 模型列表 |
| POST | /api/v1/infer | 单图推理；**query/body 须含 `image_kind`** 或 `image_id`（由 DB 带出 kind）；multipart 时与文件同发 |
| POST | /api/v1/infer/batch | 异步批推理；body 含每图 **id + image_kind** 或引用已上传 Image |
| POST | /api/v1/images:upload | 批量上传；**每文件或批次元数据须带 `image_kind`**（禁止静默默认） |
| PATCH | /api/v1/images/{id} | **可更新 `image_kind`** 等元数据；触发可选「重生成规范图」 |
| PATCH | /api/v1/images/{id}/labels | 人工标注分类 |
| POST | /api/v1/train | 提交训练作业 |
| GET | /api/v1/train/{id} | 训练状态 |
| POST | /api/v1/train/incremental | 增量训练 |
| POST | /api/v1/predictions/{id}/correct | 纠错（**annotator+admin**，viewer 禁止） |

### 7.5 异步与限流

- 长任务：**202 Accepted** + `job_id`；客户端轮询；Vue 端可封装 composable。
- CPU 场景：`429`/`503` 与队列上限；可配置 `MAX_CONCURRENT_INFER`。

## 8. 前端（Vue 3）设计

### 8.1 技术栈

| 技术 | 用途 |
|------|------|
| Vue 3 | 组合式 API（`<script setup>`） |
| Vite | 构建与本地代理 |
| Vue Router | 路由 |
| Pinia | 全局状态（用户信息、当前项目、任务轮询状态） |
| Axios / fetch | HTTP 客户端（封装 baseURL、拦截器） |
| UI 组件库 | 建议 Element Plus 或 Naive UI（待项目启动时二选一固化） |

### 8.2 开发与命名空间

- **dev**：Vite `server.proxy` 将 `/api` 转发至 FastAPI 后端（如 `http://127.0.0.1:8000`）。
- **prod**：同域由 Nginx/网关转发，或使用相对路径 `/api`。

### 8.3 登录与鉴权（前端）

- **登录页** `/login`：账号、密码、角色说明（可选）；登录成功写入 Pinia + **localStorage**（`access_token`、`user`）；进入原目标路由或 `/`。
- **路由守卫**：未登录访问受保护页 → 重定向 `/login?redirect=`；已登录访问 `/login` → `/`；**角色不足** → 跳转首页或专用 403 提示页。
- **请求拦截**：所有 `/api/*` 附带 Bearer；与 Vite `proxy` / 生产同源网关一致。

### 8.4 页面信息架构

| 路由建议 | 功能 |
|----------|------|
| /login | 账号密码登录 |
| / | 仪表盘：统计、最近任务 |
| /upload | 批量上传、重复检测提示 |
| /annotate | 标注工作台：**显示「输入类型」**；可修改分类及**更正 input type**；全脸图可展示 SAM/ROI 叠加（只读，可选） |
| /train | 发起训练、查看日志与历史 |
| /train/incremental | 增量训练向导（父模型 + 数据子集） |
| /infer | 批量推理与结果列表 |
| /review | 纠错列表：过滤低置信度预测 |
| /models | 模型注册与切换默认模型 |
| /settings | 系统设置（API Key 展示、存储路径只读等） |

### 8.5 关键交互

- **上传**：`multipart/form-data`；**每张须选择「全脸自拍 / 舌部特写」**（映射 `image_kind`）；同一批可**混合**两种类型；进度条；完成后跳转标注或推理。
- **训练日志**：SSE（若后端支持）或 2s 轮询 `GET /train/{id}`。
- **纠错**：在推理结果行打开侧栏，提交正确类名，勾选「加入再训练集」。

## 11. 安全与非功能

- 上传限制：类型白名单、单文件大小、总配额。
- 路径安全：**禁止**用户输入拼接为本地绝对路径暴露。
- 审计：训练/推理/纠错关键操作写日志（后续可加表 `audit_log`）；**登录失败**宜速率限制防爆破。

## 12. 风险与依赖决策

| 风险 | 缓解 |
|------|------|
| TongueSAM CPU 慢 | 队列 + 超时 + UI 预期；**full_face** 与 **closeup** 分队列或权重 |
| predict.py 顶层副作用 | P0 起即抽象可调用入口 |
| **image_kind 误选** | 上传强校验 + 标注页可改判 + 重生成 derived 图 |
| **混合类型训练分布漂移** | `TrainJob.params` 记录各 kind 数量；可选按 kind 分层抽样（后续） |
| 双数据库行为差异 | Alembic 双跑；避免 SQLite 特有 SQL |
| 前端与 API 版本漂移 | OpenAPI 生成 TS 类型（可选 openapi-typescript） |
| **仅前端鉴权** | 所有写操作 API 必须 `require_roles`；定期安全审查 |
